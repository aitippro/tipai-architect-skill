# 任务拆解质量检查清单

裁判拆解任务后，逐项验证。不通过的项目标记为 `⚠ 需修改`。

## 结构检查
- [ ] 任务按 Phase 1-6 分组
- [ ] Phase 顺序正确：骨架 → 组件 → 数据层 → 页面 → 集成 → 测试部署
- [ ] 技术栈一行概括完整
- [ ] 有风险提示 + 技术债务章节
- [ ] 有"执行规则"章节（5 条 Agent 执行约束）
- [ ] 验证卡含"完成时间"+"执行模型"列
- [ ] 有"执行日志"汇总模板

## 粒度检查（核心）
- [ ] 每个任务产出 ≤2 个文件
- [ ] 不存在"实现 XX 模块""搭建 XX 系统""开发 XX 功能"等 >2 文件的笼统任务
- [ ] 页面类任务至少拆为 3 个：布局骨架 / 交互逻辑 / 数据对接
- [ ] 无"完成 XX 页面"这种合并任务

## 产出明确性检查
- [ ] 每个任务在"产出文件"列有至少 1 个具体文件路径
- [ ] 文件路径不是目录（如 `src/pages/`），是精确文件名
- [ ] 配置文件类任务列出了具体文件名（如 `.eslintrc.js` 而非"lint 配置"）

## 可执行性检查
- [ ] 任务描述不含"设计""规划""决定""考虑""探索""研究"等需判断的词
- [ ] 每个任务 AI agent 拿到即可执行，无需额外设计决策
- [ ] 任务描述是祈使句式（"创建 XX 文件，内容为...""添加 XX 配置项"）

## 依赖检查
- [ ] 每个任务的依赖 ID 指向已存在的任务
- [ ] 没有循环依赖
- [ ] Phase N 不依赖 Phase N+1 的任务
- [ ] 同 Phase 内任务按依赖链线性排序（无依赖的任务在前）

## 验收检查
- [ ] 每项任务有可自动化验证的验收标准
- [ ] 验收标准不模糊（"完成 XX"不可接受，"XX 命令返回 200""页面渲染 3 个卡片"可接受）
- [ ] 涉及 API 的任务有具体 HTTP 状态码或响应断言
- [ ] 涉及 UI 的任务有具体的渲染结果描述

## 验证协议检查
- [ ] 任务清单包含"任务验证协议"章节
- [ ] 每个 Phase 末尾有验证卡表格（存在性/质量/防伪/依赖/测试/状态）
- [ ] 每个任务的验证卡有具体的检查命令（非空描述）
- [ ] 验证卡中的测试命令可实际运行（非占位符）

## 完整性检查
- [ ] Phase 1 覆盖：脚手架/包管理/目录结构/基础配置
- [ ] Phase 2 覆盖：共用组件/工具函数/类型定义/常量
- [ ] Phase 3 覆盖：DB Schema/ORM 模型/API Routes/API Client
- [ ] Phase 4 覆盖：所有页面的布局+交互+数据对接
- [ ] Phase 5 覆盖：路由/鉴权/状态管理/页面跳转
- [ ] Phase 6 覆盖：单元测试/E2E/CI 配置/部署脚本
