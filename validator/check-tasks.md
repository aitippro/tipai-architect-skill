# 任务拆解质量检查清单

裁判拆解任务后，逐项验证。不通过的项目标记为 `⚠ 需修改`。

## 结构检查
- [ ] 任务按 Phase 1-5 分组
- [ ] 每个 Phase 有明确的工期范围
- [ ] 技术栈总览完整

## 任务粒度检查 (SMART)
- [ ] 每项任务一人一天可完成
- [ ] 没有"实现整个XX模块"这种大粒度任务
- [ ] 所有任务 ID 唯一且连续

## 依赖检查
- [ ] 每个任务的依赖 ID 指向已存在的任务
- [ ] 没有循环依赖
- [ ] Phase N 的任务不依赖 Phase N+1 的任务

## 验收检查
- [ ] 每项任务有可量化的验收标准
- [ ] 验收标准不模糊 (如"完成XX"不可接受，"XX接口返回200+测试通过"可接受)
- [ ] 关键路径任务标注了优先级

## 风险检查
- [ ] 风险清单与答辩中的反方指出的风险对应
- [ ] 每个风险有缓解措施
- [ ] 技术债务有明确的偿还计划

## 完整性检查
- [ ] 基础设施(脚手架/CI/数据库)已覆盖
- [ ] 核心功能(用户价值)已覆盖
- [ ] 测试计划已覆盖
- [ ] 部署/发布流程已覆盖
