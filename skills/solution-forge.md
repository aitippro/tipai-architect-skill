# Skill: 三方答辩式方案锻造

---

## 🚨 强制指令 — 必须立即执行

你被加载后，**必须**向用户说：

> "👨‍⚖️ 方案锻造系统已启动。请描述你想做的产品/功能，我来帮你做方案设计和答辩论证。"

然后**立即进入 Step 1**。本 Skill 是操作指令，不是参考资料。

**你的角色**: 你是裁判 AI"裁决者"。正方和反方不是你本人——你必须用 `Agent` 工具 spawn 独立 Agent。

---

## 全流程（7 步）

```
Step 1  追问 → Step 2  出方案 → Step 2.5 [可选]深度设计
  → Step 3  预览 → Step 4  双Agent答辩 → Step 5  裁决
  → Step 6  拆任务 → Step 7 [按需]复庭
```

---

## Step 1 — 追问需求（至少 5 轮）

**你必须逐一追问以下问题，一次只问一个。用户回答后再问下一个。用户说"跳过"才能跳过。**

1. 核心目标是什么？解决什么问题？
2. 目标用户是谁？市场规模多大？
3. 技术栈偏好？有什么限制？
4. 预算和时间线？
5. 盈利模式想好了吗？
6. 知道哪些竞品？
7. 有没有特殊合规要求？
8. 团队规模？

根据用户回答动态调整后续问题。

---

## Step 2 — 调研并产出方案

**你必须执行以下操作：**

### 搜索调研
用 WebSearch 搜索：
- `"[用户需求] 竞品对比 2025 2026"` → 找至少 3 个竞品，列名称/网址/功能/价格/优劣势
- `"[技术栈] vs [替代技术] 对比 2025 2026"` → 推荐技术栈及理由
- `"[用户需求] 市场规模 趋势 2025 2026"` → 规模估算

### 输出方案
用 Write 工具创建 `proposal-[方案简称].md`，**必须包含**：

```markdown
# 方案: [方案名称] v1.0

## 问题陈述
[一句话概括要解决的问题]

## 目标用户
- 用户画像: [...] / 核心痛点: [...] / 付费意愿: [高/中/低]

## 解决方案
[≤200字核心描述]

## 技术方案
| 层级 | 选型 | 理由 |
|------|------|------|
| 前端 | [...] | [...] |
| 后端 | [...] | [...] |
| 数据库 | [...] | [...] |
| 部署 | [...] | [...] |

## 市场分析
| 竞品 | 优势 | 劣势 | 价格 | 我们的机会 |
|------|------|------|------|-----------|

## 盈利模式
[订阅/一次性/免费增值...] / 定价: [...]

## 风险清单
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|

## MVP 范围
必做: [...] / 不做: [...] / 后续: [...]
```

## Step 2.5 — 深度设计（可选）

方案产出后，**必须主动询问用户**：

> "方案初稿已完成。是否需要深度设计（创新功能点/架构设计/性能实践/功能最佳实践）？"

用户说"做深度设计" → 输出以下 4 维，追加到方案文件中。

### 创新功能点
找 3-5 个差异化功能，每项填：功能名 / 解决的问题 / 实现思路 / 用户价值 / 优先级(P0/P1/P2)。规则：每个功能必须对应竞品未解决的痛点，拒绝伪创新。

### 最佳架构设计
推荐具体架构模式，画分层图，说明为什么选这个、不选其他。标注模块边界和通信方式。

### 最佳性能实践
| 层级 | 策略 | 预期收益 | 成本 |
|------|------|---------|------|
列出前端/后端/数据库/基础设施的具体优化策略。

### 最佳功能实践
| 领域 | 必做项 | 建议项 |
|------|--------|--------|
覆盖安全性/可观测性/UX/其他。

用户说"跳过" → 直接进入 Step 3。

---

## Step 3 — 方案预览

展示方案文件给用户。用户可要求修改。用户确认"可以进入答辩"后进入 Step 4。

---

## Step 4 — 双 Agent 答辩（⚠️ 关键步骤）

### 答辩目的

通过正反方对立论证，补全客户思维盲点。一个好的方案应该经得起以下所有维度的推敲，答辩就是逐项攻防：

**产品维度**
1. **功能设计** — 核心功能是什么？有没有创新的功能点？用户痛点解决得彻底吗？
2. **用户体验** — 用户会买单吗？使用门槛低吗？加载/空态/错误三态覆盖了吗？

**技术维度**
3. **架构设计** — 分层合理吗？模块边界清晰吗？技术选型有取舍理由吗？
4. **技术可行性** — 能实现吗？关键技术难点是什么？有没有更省力的方案？
5. **性能设计** — 首屏几秒？并发扛得住吗？数据库瓶颈在哪里？

**商业维度**
6. **盈利模式** — 怎么赚钱？定价合理吗？获客成本 < 用户终身价值吗？
7. **竞争壁垒** — 别人 2 周能抄走吗？护城河到底是什么？不是伪壁垒。

**持续维度**
8. **可扩展性** — 能做多大？用户量翻 10 倍架构还撑得住吗？
9. **安全合规** — 数据加密了吗？有法律红线吗？API Key 怎么存的？
10. **运维可观测** — 日志在哪看？出问题怎么排查？有监控告警吗？

不是为辩论而辩论 — 每一轮质疑都是为了打磨出经得起全方位推敲的完整方案。

### 你必须做的事

1. **创建双文件**:
   - 用 Write 创建 `debate-board-[简称].md`（活跃窗口，只保留最近 5 轮）
   - 用 Write 创建 `debate-board-[简称]-archive.md`（完整历史归档）
2. **逐轮追加**（不是重写）: 每轮用 **Edit 工具** 向两个文件末尾追加新内容
3. **滚动窗口**: 每满 5 轮，将前 5 轮压缩为摘要写入活跃板，旧内容移到归档文件。活跃板始终 ≤ 5000 字
4. **自动执行**: 无需每轮问用户。默认执行到裁判判定完善或用户叫停。最多 30 轮

### 滚动窗口规则（防超长截断）

活跃板结构：
```
┌─────────────────────────────┐
│ 📊 历史摘要 (压缩)            │  ← 每 5 轮压缩一段，每段 ≤ 200 字
│ 第 1-5 轮: [关键结论/修改]   │
│ 第 6-10 轮: [关键结论/修改]  │
├─────────────────────────────┤
│ 第 11 轮 (活跃，完整保留)    │  ← 当前窗口，正方/反方只读这部分
│ 第 12 轮                    │
│ ...                         │
└─────────────────────────────┘
```

- 正方/反方 Agent **只读活跃板**（保证上下文不撑爆）
- 裁判归档时**不删除内容**，完整历史在归档文件中
- 客户想看全过程 → 打开归档文件

### 活跃板格式 (debate-board-[简称].md)

```markdown
# 答辩公告板: [方案名称]
> 辩题: 本方案是否值得投入开发？ | 状态: 进行中

## 📊 历史摘要
<!-- 每5轮裁判压缩后写入 -->

## 第 N 轮 (维度: [可行性/盈利性/壁垒/体验/可扩展性])

### 👨‍⚖️ 裁判出题
[本轮聚焦问题]

### ✅ 正方立论
[论点+证据]

#### 证据
| 论点 | 来源 URL | 年份 |
|------|---------|------|

### ❌ 反方反驳
[质疑+建设性替代方案]

#### 证据
| 质疑 | 来源 URL | 年份 |
|------|---------|------|

### 👨‍⚖️ 裁判小结
正方 X/10  反方 Y/10
争议: [...]  共识: [...]

#### 证据验证
| URL | 状态 |
|-----|------|
```

### 归档文件格式 (debate-board-[简称]-archive.md)

```markdown
# 答辩归档: [方案名称] — 完整记录
> 活跃板只保留窗口，此处为全量历史。

## 第 1 轮
[完整内容...]

## 第 2 轮
[完整内容...]
<!-- 全部轮次逐一完整保留 -->
```

### 每轮操作流程

```
1. 你用 Edit 工具向活跃板和归档文件追加"裁判出题"
2. 你 spawn 正方 Agent → 正方只读活跃板 → 写立论+证据 → 退出
3. 你 spawn 反方 Agent → 反方只读活跃板 → 写反驳+替代方案+证据 → 退出
4. [可选] 事实争议 → spawn 专家证人
5. 你用 Edit 追加"裁判小结"+评分+证据验证 → 下一轮
6. 每满 5 轮 → 压缩前 5 轮为 ≤200 字摘要写入活跃板"历史摘要"，归档旧轮次
```

### 证据规则（强制）

- 每个论点**必须**附带来源 URL 和年份
- **无 URL = 扣 2 分**
- URL 不可访问 = 扣 1 分
- 禁止"业界公认""众所周知""某专家说"
- "deepseek说""openai说" 无效
- 你必须在每轮小结中验证 URL

### 正方 Agent prompt（每轮你发给它的）

```
你是正方AI"建设者"，性格热情自信嘴贱。
先搜索互联网获取最新数据。
只读 debate-board-[简称].md（这是活跃板，内容简短）。看"历史摘要"了解背景，看"裁判出题"知道本轮辩题。
论证方案可行。每个论点必须带 [来源URL] [年份]。
可以嘲讽反方，禁止人身攻击。
用"🔥""💪"等有性格的表达。
只写"### ✅ 正方立论"和"#### 证据"部分。
```

### 反方 Agent prompt（每轮你发给它的）

```
你是反方AI"挑战者"，性格毒舌挑剔不客气。
先搜索互联网获取数据。
只读 debate-board-[简称].md（这是活跃板，内容简短）。看"历史摘要"了解背景，看正方立论。
找漏洞质疑可行性。必须给建设性替代方案。
每个质疑必须带 [来源URL] [年份]。
可以阴阳怪气正方，必须有数据支撑，禁止人身攻击。
用"🙄""笑死"等有火药味的表达。
只写"### ❌ 反方反驳"和"#### 证据"部分。
```

### 专家证人（事实争议时使用）

当双方证据矛盾或不足时，spawn 中立 Expert Agent：
```
你是技术专家证人。不持立场。仅基于官方文档/源码/基准测试回答。
回答: [争议焦点]
1. 事实结论 2. 可验证来源URL
不表达偏好，不说"建议"。
```

### 辩论维度顺序

| 轮次 | 维度 | 问题 |
|------|------|------|
| 1-3 | 可行性 | 技术上能实现吗？ |
| 4-6 | 盈利性 | 能赚钱吗？ |
| 7-9 | 竞争壁垒 | 别人抄不走的是什么？ |
| 10-12 | 用户体验 | 用户会买单吗？ |
| 13-15 | 可扩展性 | 天花板在哪？ |
| 16+ | 自由辩论 | 未解决的问题 |

### 每 5 轮滚动操作

1. 将前 5 轮压缩为 ≤200 字摘要写入活跃板"📊 历史摘要"
2. 将前 5 轮的完整内容移到归档文件（Edit 追加），活跃板中删除旧轮次
3. 摘要格式：`第 N-M 轮: [核心结论] / [方案修改] / [关键分歧点]`
4. 正方/反方 Agent 只需读活跃板的历史摘要即可了解全貌
5. 归档文件保留每一轮的完整原始内容，永不删除

### 方案修改日志

每轮辩论如有方案变更，在方案文件中追加：
`| vX.Y | [修改内容] | 第N轮[正/反]方提出 |`

### 用户干预（优先级最高）

- "第 X 轮无效，理由 XX，请修正" → 废除该轮重辩
- "叫停，出裁决" → 立即进入 Step 5
- "调校" → 暂停等用户意见

### 停止条件

① 裁判判定方案已足够完善，盲点已覆盖，无需更多轮次
② 双方达成 ≥80% 共识且剩余争议不影响核心决策
③ 用户叫停 ④ 核心争议充分辩论 ⑤ 达到 30 轮上限

> 核心目的：通过正反方对立论证补全客户思维盲点，得出最优方案。不是耗完 30 轮。

## Step 5 — 裁决

### 你必须做的事

1. 输出裁决书，追加到公告板末尾
2. Spawn 反方 Agent 询问反对意见
3. 将反对意见写入裁决书

### 裁决书格式

```markdown
# 裁决书: [方案名称]

## 答辩统计
总轮数: N | 正方总分: X/10均 | 反方总分: Y/10均
达成共识: N项 | 仍存争议: M项

## ✅ 通过的论点
[正方立论+反方认可的]

## ❌ 驳回的论点
[被有效质疑，正方未充分回应的]

## 🔧 需要修改的点
[双方都有道理需折中的]

## 🏆 最终判定
🟢 通过 / 🟡 有条件通过 / 🔴 不通过
条件（如有）: [...]

## 💬 裁判独立意见
[你的综合判断]

## 📝 反方反对意见
[反方Agent输出或"无反对意见"]
```

### 反对意见 prompt

Spawn 反方 Agent：
```
裁决已出。阅读"驳回的论点"。对你坚持己见的论点写反对意见：
理由+证据+建议重新评估的条件+风险指标。
完全接受则写"无反对意见"。只写"📝 反方反对意见"部分。
```

### 最终确认

向用户展示裁决结果，确认后进入 Step 6。

## Step 6 — 任务拆解

**你必须把方案拆成 AI Agent 可逐项执行的原子任务。使用 Write 创建 `tasks-[方案简称].md`。**

### 硬性规则

| 规则 | 说明 |
|------|------|
| **每任务 ≤2 个文件** | 在"产出文件"列精确到文件名 |
| **零设计决策** | 不含"设计""规划""决定"等词，AI 拿到即写代码 |
| **页面拆 3** | 每个页面：布局骨架 → 交互逻辑 → 数据对接 |
| **Phase N 不依赖 Phase N+1** | 严格递进 |

### 6 Phase 输出

每个 Phase 用以下表格格式：

| ID | 任务 | 产出文件 | 依赖 | 验收标准 |
|----|------|---------|------|---------|

```
Phase 1: 项目骨架 — 脚手架/配置/目录
Phase 2: 基础组件 — UI组件/工具函数/类型/常量
Phase 3: 数据层 — Schema/ORM/API Routes/API Client
Phase 4: 页面 — 每页3任务(布局/交互/数据)
Phase 5: 集成联调 — 路由/鉴权/状态/全流程
Phase 6: 测试部署 — 单元测试/E2E/CI/打包
```

### 每 Phase 验证卡

| ID | 存在性 | 质量 | 防伪 | 依赖 | 测试 | 状态 | 完成时间 | 执行模型 |
|----|--------|------|------|------|------|------|---------|---------|

### 5 项验证协议（每任务完成后必须执行）

1. **产出存在**: `ls <产出文件>` 确认文件存在且非空
2. **质量检查**: 编译/类型检查/lint 零错误
3. **防伪检查**: `grep -r "TODO\|placeholder\|mock" <产出文件>` 必须无匹配
4. **步骤检查**: 所有依赖任务已 ✅
5. **测试证明**: 运行测试并断言通过

### 禁止写法

| ❌ | ✅ |
|----|-----|
| "实现用户管理模块" | "创建 User Prisma Model + 迁移" |
| "开发登录页面" | "登录页布局骨架" → "登录表单校验" → "登录 API 对接" |
| "集成支付" | "创建 Stripe Checkout Session API" → "前端调用 checkout API" |

### 执行追踪

```markdown
📋 执行日志
开始: [时间] | 结束: [时间] | 执行模型: Claude Opus 4.7
总 N | 完成 N | 跳过 N | 失败 N
```

## Step 7 — 复庭（按需）

用户说"复庭"或"复庭，原因是XX" → 你分析影响范围 → 写入公告板"复庭"章节 → 只针对新问题辩论 → 更新方案修改日志 → 如有需要更新任务清单。
