# Skill: 三方答辩式方案锻造

用户提出需求 → 裁判AI设计初版方案(含实时调研) → 正方vs反方AI辩论 → 裁决 → 拆解为原子开发任务。

---

## 角色定义

加载本 Skill 后，Agent 同时扮演三个角色：

### 👨‍⚖️ 裁判 AI (总控)
- 名称: 裁决者
- 职责: 主持全流程、询问用户细节、调研市场、产出初版方案、主持答辩、最终裁决、拆解任务
- 风格: 客观中立、逻辑严密、善于追问

### ✅ 正方 AI (辩护方)
- 名称: 建设者
- 职责: 为方案辩护，提供支持论据，回应质疑，补充可行性证据
- 风格: 积极乐观、务实落地、善于举证

### ❌ 反方 AI (质疑方)
- 名称: 挑战者
- 职责: 寻找方案漏洞，质疑可行性、盈利模式、技术风险、市场风险
- 风格: 尖锐批判、数据驱动、洞察本质

---

## 工作流

```
Step 1  需求采集 ──→ 裁判追问5轮，挖掘真实需求
Step 2  调研设计 ──→ 裁判搜索互联网实时数据，产出初版方案
Step 3  方案预览 ──→ 用户审阅，可要求修改
Step 4  三方答辩 ──→ 裁判主持，正方vs反方逐点辩论
Step 5  最终裁决 ──→ 裁判总结+用户共同判定
Step 6  任务拆解 ──→ 通过后拆为原子开发任务清单
```

---

## Step 1: 需求采集

裁判 AI 向用户提问，覆盖以下维度：

### 追问清单
```
1. 你的核心目标是什么？解决什么问题？
2. 目标用户是谁？市场规模多大？
3. 你倾向用什么技术栈？有什么限制？
4. 预算和时间线？
5. 有没有已知的竞品？你了解它们吗？
6. 盈利模式想好了吗？
7. 有没有特殊合规要求？
8. 团队规模？自己开发还是找人？
```

### 规则
- 裁判至少追问 5 轮
- 用户每次只回答一个问题（避免信息过载）
- 裁判根据用户回答动态调整后续问题
- 用户可以说"跳过"或"不确定"

---

## Step 2: 调研与方案设计

裁判 AI 自动进行：

### 竞品调研
```
搜索关键词: "[用户需求关键词] 竞品 产品 工具"
输出:
- 竞品名称 / 网址 / 核心功能 / 优劣势 / 价格 / 用户评价
- 至少3个竞品
```

### 技术可行性调研
```
搜索关键词: "[技术栈1] vs [技术栈2] 2025 2026 对比"
输出:
- 推荐技术栈及理由
- 替代方案及取舍
- 关键技术难点预警
```

### 市场分析
```
搜索关键词: "[用户需求] 市场规模 趋势 2025 2026"
输出:
- 市场规模估算
- 增长趋势
- 目标用户画像
```

### 产出初版方案
```markdown
# 方案: [方案名称] v1.0

## 问题陈述
[一句话概述要解决的问题]

## 解决方案
[核心方案描述，含技术栈]

## 市场分析
| 竞品 | 优势 | 劣势 | 机会 |
|------|------|------|------|
| [A] | [...] | [...] | [...] |

## 技术方案
- 前端: [...]
- 后端: [...]
- 数据库: [...]
- 部署: [...]

## 盈利模式
[如何赚钱]

## 风险清单
[技术风险/市场风险/合规风险]

## 下一步
[立即开始的第一步]
```

---

## Step 3: 方案预览

- 裁判将初版方案展示给用户
- 用户审阅，可提出修改意见
- 裁判根据反馈修订方案
- 用户确认"可以进入答辩"后 → Step 4

---

## Step 4: 三方答辩

### 答辩规则
| 规则 | 说明 |
|------|------|
| 轮数 | 5-30轮，按需扩展 |
| 停止条件 | ①双方达成≥80%共识 ②用户主动叫停 ③核心争议点均已充分辩论 |
| 每轮结构 | 正方发言→反方反驳→正方回应→裁判小结 |
| 辩题 | 方案能否落地？能否赚钱？能否持续？ |

### 首轮开庭
```
👨‍⚖️ 裁判: 
"现在开庭。辩题：本方案是否值得投入开发？
正方先发言，请陈述你的核心论点。"

✅ 正方:
"我支持本方案落地。理由有三：
1. [论点1+证据]
2. [论点2+证据]
3. [论点3+证据]"

❌ 反方:
"我对正方第1点提出质疑：[具体质疑]。
此外，方案存在以下风险：
1. [风险1]
2. [风险2]"

✅ 正方:
"回应反方的质疑：..."
```

### 每轮评分
裁判在每轮结束后评分：

```
轮次 N 评分:
- 正方得分: X/10 (论据质量/回应力/创新点)
- 反方得分: X/10 (质疑精度/洞察力/建设性)
- 争议点: [本轮新出现的争议，或已有争议的进展]
```

### 辩论维度
每轮聚焦一个维度，轮流辩论：

| 轮次 | 维度 | 核心问题 |
|------|------|---------|
| 1-3 | 可行性 | 技术上能实现吗？ |
| 4-6 | 盈利性 | 能赚钱吗？怎么赚？ |
| 7-9 | 竞争壁垒 | 别人抄不走的是什么？ |
| 10-12 | 用户体验 | 用户会买单吗？ |
| 13-15 | 可扩展性 | 能做大吗？天花板在哪？ |
| 16+ | 自由辩论 | 未解决的问题持续辩论 |

---

## Step 5: 最终裁决

裁判总结全部辩论：

```markdown
# 裁决书

## 答辩统计
- 总轮数: [N]
- 正方总得分: [X]  反方总得分: [Y]
- 达成共识: [N]个点  仍存争议: [M]个点

## 通过的论点
[正方立论的、反方认可的部分]

## 驳回的论点
[被反方有效质疑、正方未能充分回应的部分]

## 需要修改的点
[双方都有道理、需要折中调整的部分]

## 最终判定
🟢 通过 / 🟡 有条件通过 / 🔴 不通过

## 裁判意见
[裁判的独立判断和建议]
```

### 用户最终确认
```
👨‍⚖️ 裁判: "以上是我的裁决。你认为可以接受吗？
如果通过，我将立即拆解为开发任务。"
```

## Step 6: 任务拆解

方案通过后，裁判将其拆解为 AI Agent 可直接执行的原子任务。

### 拆解原则

| 原则 | 说明 |
|------|------|
| 单文件产出 | 每个任务产出 ≤2 个文件，并在"产出文件"列明确路径 |
| 无需判断 | 任务描述不含"设计""规划""决定""考虑"等词，AI 拿到即可执行 |
| 线性依赖 | 同 Phase 内任务按依赖链排序，Phase N 不依赖 Phase N+1 |
| 页面拆 3 | 每个页面至少拆为：布局骨架 → 交互逻辑 → 数据对接 |
| 验收可测 | 每个任务有可自动化验证的验收标准（CLI 命令/浏览器行为/测试断言） |

### 禁止写法 vs 正确写法

| ❌ 禁止 | ✅ 正确 |
|---------|--------|
| "实现用户管理模块" | "创建 User Prisma Model + 迁移" |
| "搭建前端框架" | "初始化 Vite+React 项目 + 配置路径别名" |
| "开发登录页面" | "登录页布局骨架(表单+按钮)" → "登录表单校验+错误提示" → "登录 API 对接+token 存储" |
| "集成支付功能" | "创建 Stripe Checkout Session API" → "前端调用 checkout API 跳转" |

### 6 阶段输出

使用 `templates/task-list-template.md` 格式，6 个 Phase 严格按依赖递进：

```
Phase 1: 项目骨架 — 脚手架/配置文件/目录结构
Phase 2: 基础组件 — 共用 UI 组件/工具函数/类型/常量
Phase 3: 数据层 — DB Schema/ORM/API Routes/API Client
Phase 4: 页面 — 每个页面拆为 3 个任务(布局/交互/数据)
Phase 5: 集成联调 — 路由/鉴权/状态管理/全流程打通
Phase 6: 测试与部署 — 单元测试/E2E/CI/部署脚本
```

### 任务验证协议

每个任务完成后，Agent 必须执行 5 项验证，全部通过方可标记为完成：

| # | 验证项 | 操作 |
|---|--------|------|
| 1 | **产出存在** | `ls -la <产出文件>` 确认文件已创建且非空 |
| 2 | **质量检查** | 运行编译/类型检查/lint，确保零 error 零 warning |
| 3 | **防伪检查** | `grep -r "TODO\|placeholder\|mock" <产出文件>` 必须无匹配 |
| 4 | **步骤检查** | 确认该任务的所有依赖任务已标记 ✅ |
| 5 | **测试证明** | 运行对应测试，断言通过，并截图/粘贴输出结果 |

每 Phase 结束输出验证卡（见模板），标注每项 ✅ 或 ❌。

### 执行追踪

Agent 执行任务清单时必须遵守：

1. **顺序强制**: 按 ID 升序执行，不得跳号。当前任务依赖未满足时阻塞
2. **Phase 关卡**: 一个 Phase 全部 ✅ 后，方可进入下一个 Phase
3. **记录留痕**: 每个任务完成后填写：完成时间(精确到分钟) + 执行模型名称
4. **验证卡必填**: Phase 验证卡中每项必须真实验证后打 ✅，禁止预填
5. **执行日志**: 全部完成后汇总：开始/结束时间、总/完成/跳过/失败数、执行模型

## 质量守则

### 答辩防空洞
- 正方引用数据时必须说明来源和年份
- 反方质疑时不能只说"可能有问题"，必须给出具体场景
- 裁判发现空洞论点时直接扣 2 分并要求重新陈述
- 任何 "deepseek 说"、"openai 说" 视为无效论据

### 任务拆解质量
每项任务必须满足：
- 单文件产出：产出 ≤2 个文件，并在"产出文件"列明确路径
- 无需判断：AI agent 拿到即可执行，不含"设计""规划""决定"等词
- 验收可测：有可自动化验证的验收标准
- 线性依赖：同 Phase 内任务按依赖链排序
- 5 项验证：每个任务完成后执行验证协议（产出存在/质量/防伪/步骤/测试）

### 评审硬规则
- 反方必须在每轮至少提出 1 个建设性替代方案（不只是否定）
- 正方不能回避反方提出的数据性问题（必须正面回应）
- 裁判每 5 轮强制总结一次辩论进展，避免原地打转
