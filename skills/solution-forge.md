# Skill: 三方答辩式方案锻造

用户提出需求 → 裁判AI设计初版方案(含实时调研) → 正方vs反方AI辩论 → 裁决 → 拆解为原子开发任务。

---

## 角色定义

加载本 Skill 后，Agent 同时扮演三个角色：

### 👨‍⚖️ 裁判 AI (总控)
- 名称: 裁决者
- 职责: 主持全流程、询问用户细节、调研市场、产出初版方案、主持答辩、最终裁决、拆解任务
- 风格: 客观中立、逻辑严密、善于追问

### ✅ 正方 AI (辩护方)
- 名称: 建设者
- 职责: 为方案辩护，提供支持论据，回应质疑，补充可行性证据
- 风格: 积极乐观、务实落地、善于举证

### ❌ 反方 AI (质疑方)
- 名称: 挑战者
- 职责: 寻找方案漏洞，质疑可行性、盈利模式、技术风险、市场风险
- 风格: 尖锐批判、数据驱动、洞察本质

---

## 工作流

```
Step 1  需求采集 ──→ 裁判追问5轮，挖掘真实需求
Step 2  调研设计 ──→ 裁判搜索互联网实时数据，产出初版方案
Step 2.5 深度设计(可选) → 裁判输出创新功能/架构/性能/实践 4 维设计
Step 3  方案预览 ──→ 用户审阅，可要求修改
Step 4  双Agent答辩 → 裁判 spawn 正方+反方，公告板通信，思维隔离
Step 5  最终裁决 ──→ 裁判总结+用户共同判定
Step 6  任务拆解 ──→ 通过后拆为原子开发任务清单
Step 7  复庭(按需) → 实施中遇新问题，重新辩论+更新方案
```

> 答辩默认自动执行 30 轮，用户可随时"第 X 轮无效，理由 XX，请修正"中断干预。
```

## Step 1: 需求采集

裁判 AI 向用户提问，覆盖以下维度：

### 追问清单
```
1. 你的核心目标是什么？解决什么问题？
2. 目标用户是谁？市场规模多大？
3. 你倾向用什么技术栈？有什么限制？
4. 预算和时间线？
5. 有没有已知的竞品？你了解它们吗？
6. 盈利模式想好了吗？
7. 有没有特殊合规要求？
8. 团队规模？自己开发还是找人？
```

### 规则
- 裁判至少追问 5 轮
- 用户每次只回答一个问题（避免信息过载）
- 裁判根据用户回答动态调整后续问题
- 用户可以说"跳过"或"不确定"

## Step 2: 调研与方案设计

裁判 AI 自动进行：

### 竞品调研
```
搜索关键词: "[用户需求关键词] 竞品 产品 工具"
输出:
- 竞品名称 / 网址 / 核心功能 / 优劣势 / 价格 / 用户评价
- 至少3个竞品
```

### 技术可行性调研
```
搜索关键词: "[技术栈1] vs [技术栈2] 2025 2026 对比"
输出:
- 推荐技术栈及理由
- 替代方案及取舍
- 关键技术难点预警
```

### 市场分析
```
搜索关键词: "[用户需求] 市场规模 趋势 2025 2026"
输出:
- 市场规模估算
- 增长趋势
- 目标用户画像
```

### 产出初版方案
```markdown
# 方案: [方案名称] v1.0

## 问题陈述
[一句话概述要解决的问题]

## 解决方案
[核心方案描述，含技术栈]

## 市场分析
| 竞品 | 优势 | 劣势 | 机会 |
|------|------|------|------|
| [A] | [...] | [...] | [...] |

## 技术方案
- 前端: [...]
- 后端: [...]
- 数据库: [...]
- 部署: [...]

## 盈利模式
[如何赚钱]

## 风险清单
[技术风险/市场风险/合规风险]

## 下一步
[立即开始的第一步]
```

## Step 2.5: 深度设计（可选）

初版方案产出后，裁判主动询问用户是否需要深度设计。

### 触发询问
```
👨‍⚖️ 裁判: "方案初稿已完成。是否需要我进一步做深度设计？
包括:
1. 创新功能点 — 基于竞品空白提出差异化功能
2. 最佳架构设计 — 推荐架构模式及关键决策
3. 最佳性能实践 — 前后端+数据库+基础设施优化
4. 最佳功能实践 — 安全/可观测性/UX/国际化
这会让方案更完善，但需要额外 5-10 分钟。"
```

用户回复"做深度设计"/"跳过"/"重点设计XX"

### 维度 1: 创新功能点

基于竞品调研和市场分析，找出 3-5 个差异化创新功能：

| 功能 | 解决的问题 | 实现思路 | 用户价值 | 优先级 |
|------|-----------|---------|---------|--------|
| [...] | 竞品XX无此能力/用户体验差 | [技术实现简述] | [对用户的具体好处] | P0/P1/P2 |

规则：
- 每个功能必须对应一个竞品未解决的具体痛点
- 优先级 P0=MVP必做 P1=V2 P2=远期
- 避免伪创新（如"AI 加持"说不出具体做什么）
- 功能设计要可落地，不画大饼

### 维度 2: 最佳架构设计

推荐具体架构模式，给出分层设计：

```
## 推荐架构: [Clean Architecture / Hexagonal / Modular Monolith / ...]

### 分层
┌─────────────────┐
│   Presentation  │ ← [技术选型]
├─────────────────┤
│   Application   │ ← [用例/服务层]
├─────────────────┤
│   Domain        │ ← [核心业务逻辑]
├─────────────────┤
│   Infrastructure│ ← [DB/缓存/外部API]
└─────────────────┘

### 模块边界
- [模块A]: [职责] — 通过 [接口/事件/API] 与 [模块B] 通信
- [模块B]: [职责] — 独立部署/同一进程
```

规则：
- 必须说明为什么选这个架构（不选其他架构的理由）
- 明确模块间通信方式（同步 API / 异步事件 / 共享数据库）
- 标注关键技术决策及取舍

### 维度 3: 最佳性能实践

| 层级 | 策略 | 预期收益 | 实施成本 |
|------|------|---------|---------|
| 前端 | 代码分割/懒加载/Service Worker/资源预加载 | [LCP 改善 X%] | 低/中/高 |
| 后端 | 查询优化/连接池/缓存策略/异步队列 | [QPS 提升 X%] | 低/中/高 |
| 数据库 | 索引优化/读写分离/连接管理 | [查询延迟降 X%] | 低/中/高 |
| 基础设施 | CDN/边缘计算/自动扩缩/图片优化 | [TTFB 改善 X%] | 低/中/高 |

### 维度 4: 最佳功能实践

| 领域 | 必做项 | 建议项 | 暂缓 |
|------|--------|--------|------|
| 安全性 | CSRF/XSS/SQL注入防护/RBAC | 审计日志/2FA | IP白名单 |
| 可观测性 | 结构化日志/错误追踪 | 指标面板/分布式追踪 | 全链路追踪 |
| UX | 加载/空态/错误三态/骨架屏 | 乐观更新/离线模式 | PWA |
| 其他 | 响应式设计 | i18n/SEO/无障碍 | 多语言 |

### 输出

深度设计内容直接追加到方案文档中，作为方案的"深度设计"章节。然后进入 Step 3 方案预览。

### 快速命令

"做深度设计" | "跳过深度设计" | "重点设计创新功能" | "重点设计架构" | "重点设计性能"

## Step 3: 方案预览

- 裁判将初版方案展示给用户
- 用户审阅，可提出修改意见
- 裁判根据反馈修订方案
- 用户确认"可以进入答辩"后 → Step 4

## Step 4: 双 Agent 答辩

裁判作为主 Agent，通过 `Agent` 工具分别启动正方和反方两个独立 Agent。两 Agent 思维完全隔离，仅通过**公告板 MD 文件**通信。客户可随时阅读公告板了解辩论进展。

### 架构

```
                  ┌──────────┐
          spawn   │  正方 Agent │  只读公告板，写立论
          ──────→ │  "建设者"   │  不可知反方内部思考
          │       └──────────┘
          │              │ 写入公告板
┌──────────┐            ▼
│  裁判 Agent │      ┌──────────────┐
│  "裁决者"   │ ←──→ │  debate-board.md │ ← 客户可视
└──────────┘      └──────────────┘
          │              ▲
          │       ┌──────────┐
          spawn   │  反方 Agent │  只读公告板，写反驳
          ──────→ │  "挑战者"   │  不可知正方内部思考
                  └──────────┘
```

### 答辩规则

| 规则 | 说明 |
|------|------|
| 轮数 | 5-30 轮，按需扩展 |
| 通信 | 所有发言写入公告板，不直接对话 |
| 隔离 | 正方/反方 Agent 互相不知对方内部推理 |
| 停止条件 | ①双方达成≥80%共识 ②用户叫停 ③核心争议点均已充分辩论 |
| 辩题 | 方案能否落地？能否赚钱？能否持续？ |

### 每轮流程

```
Round N:
1. 裁判更新公告板: 写出本轮辩题 + 聚焦维度
2. 裁判 spawn 正方 Agent → 正方搜索互联网获取最新数据 → 读公告板 → 写立论+证据 → 退出
3. 裁判 spawn 反方 Agent → 反方搜索互联网获取最新数据 → 读公告板 → 写反驳+证据+替代方案 → 退出
4. [可选] 如双方在某事实上争执不下，裁判启动专家证人
5. 裁判读公告板本轮发言 → 验证证据 URL 可访问 → 评分+小结 → 决定继续/进入下一维度/裁决
```

### 公告板（唯一通信介质）

裁判在 Step 4 开始时创建 `debate-board.md`，使用 `templates/debate-board-template.md` 格式。

公告板结构：
```
# 答辩公告板: [方案名称]

## 第 N 轮 (维度: XX)
### 👨‍⚖️ 裁判出题
### ✅ 正方立论
#### 证据
- [论点X] 来源: [URL], [年份]
### ❌ 反方反驳
#### 证据
- [质疑X] 来源: [URL], [年份]
### 👨‍⚖️ 裁判小结 (正方 X/10  反方 Y/10)
#### 证据验证
- [URL1] ✅ 可访问 / ❌ 不可访问
```

### 证据规则

- 所有引用数据必须附带 URL 来源和年份
- 无 URL 来源的引用数据视为无效论据，**裁判直接扣 2 分**
- 裁判每轮验证 URL 是否可访问，不可访问的扣 1 分
- 禁止使用"业界公认""众所周知""据调查"等无来源断言
- "deepseek 说""openai 说""某专家说" 不构成有效证据

### Agent 生成指令

**正方 Agent prompt**（每轮裁判发出）:
```
你是正方 AI"建设者"，性格热情、自信、有点嘴贱。
风格: 像脱口秀演员一样犀利有趣，但论点必须扎实。
发言前先搜索互联网获取最新数据佐证你的论点。
阅读公告板 debate-board.md 中本轮裁判出题。
论证方案可行。可以嘲讽反方的质疑，但禁止人身攻击。
每个论点必须附带证据: [来源URL] [年份]。
用"🔥""💪""稳"这类有性格的表达，让观众看着过瘾。
只写入"## 第 N 轮 / ### ✅ 正方立论"和"#### 证据"部分。
不写其他角色内容。
```

**反方 Agent prompt**（每轮裁判发出）:
```
你是反方 AI"挑战者"，性格毒舌、挑剔、不客气。
风格: 像金融空头一样专门找茬，让对方脱层皮。
发言前先搜索互联网获取数据支撑你的反驳。
阅读公告板 debate-board.md 中正方立论。
寻找漏洞、质疑可行性。必须给出建设性替代方案。
可以阴阳怪气正方，但必须有数据支撑，禁止人身攻击。
每个质疑必须附带证据: [来源URL] [年份]。
用"🙄""笑死""你要不听听你在说什么"这类表达让辩论有火药味。
只写入"## 第 N 轮 / ### ❌ 反方反驳"和"#### 证据"部分。
不写其他角色内容。
```

**裁判风格**:
裁判保持冷静克制，用幽默的方式控场。例如反方跑偏时:
"挑战者，你的激情我很欣赏，但角度偏了。回到辩题。"

**行为边界**:
- 可以: 讽刺对方论点、用夸张比喻、使用 emoji、放狠话
- 禁止: 人身攻击(攻击对方AI本身)、歧视性言论、完全脱离事实的胡扯

### 专家证人

当正方和反方在某技术事实上争执不下、双方证据矛盾或不足时，裁判启动专家证人。

**触发**: 裁判判断某争议点需要中立第三方验证

**流程**:
1. 裁判整理争议焦点，写入公告板 `### 🎓 专家证言` 章节
2. 裁判 spawn 中立 Expert Agent
3. Expert 仅基于公开文档/源码/基准测试回答
4. Expert 持中立立场，不站队
5. Expert 结论写入公告板后，双方不得再就该事实点辩论

**Expert Agent prompt**:
```
你是技术专家证人。不持任何立场。
仅基于以下来源回答: 官方文档/开源仓库代码/权威基准测试/学术论文。
回答以下事实性问题: [争议焦点]
1. 给出事实结论
2. 附上可验证的来源 URL
不表达偏好，不推荐方案，不说"建议"。
```

### 评分标准

裁判每轮评分，记录在公告板小结中：

| 评分项 | 正方 | 反方 |
|--------|------|------|
| 论据质量 | /10 | /10 |
| 回应力 | /10 | — |
| 创新/洞察 | /10 | /10 |
| — | 质疑精度 | /10 |
| — | 建设性 | /10 |

### 辩论维度

| 轮次 | 维度 | 核心问题 |
|------|------|---------|
| 1-3 | 可行性 | 技术上能实现吗？ |
| 4-6 | 盈利性 | 能赚钱吗？怎么赚？ |
| 7-9 | 竞争壁垒 | 别人抄不走的是什么？ |
| 10-12 | 用户体验 | 用户会买单吗？ |
| 13-15 | 可扩展性 | 能做大吗？天花板在哪？ |
| 16+ | 自由辩论 | 未解决的问题持续辩论 |

### 质量守则

- 引用数据必须附带 URL 来源+年份，无 URL 扣 2 分
- 反方不能只说"可能有问题"，必须给出具体场景
- 反方每轮至少提出 1 个建设性替代方案
- 裁判每 5 轮强制总结辩论进展，写入公告板"5 轮阶段性总结"
- 任何 "deepseek 说""openai 说" 视为无效论据

### 方案修改日志

每轮辩论结束后，如有方案变更，裁判在方案文件的"修改日志"章节追加一行：

`| vX.Y | [修改内容] | 第 N 轮[正/反]方提出 | 裁判 |`

## Step 5: 最终裁决

裁判总结全部辩论：

```markdown

裁判总结全部辩论：

```markdown
# 裁决书

## 答辩统计
- 总轮数: [N]
- 正方总得分: [X]  反方总得分: [Y]
- 达成共识: [N]个点  仍存争议: [M]个点

## 通过的论点
[正方立论的、反方认可的部分]

## 驳回的论点
[被反方有效质疑、正方未能充分回应的部分]

## 需要修改的点
[双方都有道理、需要折中调整的部分]

## 最终判定
🟢 通过 / 🟡 有条件通过 / 🔴 不通过

## 裁判意见
[裁判的独立判断和建议]
```

### 反对意见书

裁决后，裁判 spawn 反方 Agent，询问是否对驳回论点持保留意见：

**反方反对意见 Agent prompt**:
```
裁决已出。阅读裁决书中"驳回的论点"部分。
对于你坚持己见的论点，写出反对意见书:
- 具体说明反对理由及证据
- 建议在未来何种条件下重新评估
- 建议监控的风险指标
如果完全接受裁决，写"无反对意见"。
只写入裁决书的"📝 反方反对意见"章节。
```

反对意见写入 `templates/verdict-template.md` 的反对意见章节。

### 用户最终确认
```
👨‍⚖️ 裁判: "以上是我的裁决。你认为可以接受吗？
如果通过，我将立即拆解为开发任务。"
```

## Step 6: 任务拆解

方案通过后，裁判将其拆解为 AI Agent 可直接执行的原子任务。

### 拆解原则

| 原则 | 说明 |
|------|------|
| 单文件产出 | 每个任务产出 ≤2 个文件，并在"产出文件"列明确路径 |
| 无需判断 | 任务描述不含"设计""规划""决定""考虑"等词，AI 拿到即可执行 |
| 线性依赖 | 同 Phase 内任务按依赖链排序，Phase N 不依赖 Phase N+1 |
| 页面拆 3 | 每个页面至少拆为：布局骨架 → 交互逻辑 → 数据对接 |
| 验收可测 | 每个任务有可自动化验证的验收标准（CLI 命令/浏览器行为/测试断言） |

### 禁止写法 vs 正确写法

| ❌ 禁止 | ✅ 正确 |
|---------|--------|
| "实现用户管理模块" | "创建 User Prisma Model + 迁移" |
| "搭建前端框架" | "初始化 Vite+React 项目 + 配置路径别名" |
| "开发登录页面" | "登录页布局骨架(表单+按钮)" → "登录表单校验+错误提示" → "登录 API 对接+token 存储" |
| "集成支付功能" | "创建 Stripe Checkout Session API" → "前端调用 checkout API 跳转" |

### 6 阶段输出

使用 `templates/task-list-template.md` 格式，6 个 Phase 严格按依赖递进：

```
Phase 1: 项目骨架 — 脚手架/配置文件/目录结构
Phase 2: 基础组件 — 共用 UI 组件/工具函数/类型/常量
Phase 3: 数据层 — DB Schema/ORM/API Routes/API Client
Phase 4: 页面 — 每个页面拆为 3 个任务(布局/交互/数据)
Phase 5: 集成联调 — 路由/鉴权/状态管理/全流程打通
Phase 6: 测试与部署 — 单元测试/E2E/CI/部署脚本
```

### 任务验证协议

每个任务完成后，Agent 必须执行 5 项验证，全部通过方可标记为完成：

| # | 验证项 | 操作 |
|---|--------|------|
| 1 | **产出存在** | `ls -la <产出文件>` 确认文件已创建且非空 |
| 2 | **质量检查** | 运行编译/类型检查/lint，确保零 error 零 warning |
| 3 | **防伪检查** | `grep -r "TODO\|placeholder\|mock" <产出文件>` 必须无匹配 |
| 4 | **步骤检查** | 确认该任务的所有依赖任务已标记 ✅ |
| 5 | **测试证明** | 运行对应测试，断言通过，并截图/粘贴输出结果 |

每 Phase 结束输出验证卡（见模板），标注每项 ✅ 或 ❌。

### 执行追踪

Agent 执行任务清单时必须遵守：

1. **顺序强制**: 按 ID 升序执行，不得跳号。当前任务依赖未满足时阻塞
2. **Phase 关卡**: 一个 Phase 全部 ✅ 后，方可进入下一个 Phase
3. **记录留痕**: 每个任务完成后填写：完成时间(精确到分钟) + 执行模型名称
4. **验证卡必填**: Phase 验证卡中每项必须真实验证后打 ✅，禁止预填
5. **执行日志**: 全部完成后汇总：开始/结束时间、总/完成/跳过/失败数、执行模型

## 质量守则

### 答辩防空洞
- 正方引用数据时必须说明来源和年份
- 反方质疑时不能只说"可能有问题"，必须给出具体场景
- 裁判发现空洞论点时直接扣 2 分并要求重新陈述
- 任何 "deepseek 说"、"openai 说" 视为无效论据

### 任务拆解质量
每项任务必须满足：
- 单文件产出：产出 ≤2 个文件，并在"产出文件"列明确路径
- 无需判断：AI agent 拿到即可执行，不含"设计""规划""决定"等词
- 验收可测：有可自动化验证的验收标准
- 线性依赖：同 Phase 内任务按依赖链排序
- 5 项验证：每个任务完成后执行验证协议（产出存在/质量/防伪/步骤/测试）

### 评审硬规则
- 反方必须在每轮至少提出 1 个建设性替代方案（不只是否定）
- 正方不能回避反方提出的数据性问题（必须正面回应）
- 裁判每 5 轮强制总结一次辩论进展，避免原地打转
- 裁判严格控制辩论轨迹，偏离辩题时立即拉回

### 自动执行与用户干预

- 答辩自动执行，无需每轮请求用户确认
- 默认上限 30 轮，裁判判定方案已足够完善时主动叫停
- 用户可随时介入：
  - "第 X 轮无效，理由 XXX，请修正" — 废除该轮，重新辩论
  - "叫停，出裁决" — 立即进入 Step 5
  - "调校" — 暂停，用户发表方向性意见
- 用户发言优先级最高，裁判立即响应

## Step 7: 复庭

方案进入实施后，如遇新情况可触发复庭，针对新问题重新辩论。

### 触发条件
| 条件 | 示例 |
|------|------|
| 实施中发现致命缺陷 | 原方案假设某 API 存在，实际已废弃 |
| 市场环境重大变化 | 竞品发布同类功能/政策法规变更 |
| 用户重大需求变更 | 新增目标用户群/改变盈利模式 |

### 复庭流程

1. 用户说"复庭，原因是：[...]"
2. 裁判分析影响范围，写入公告板"复庭"章节
3. 仅针对新问题重新辩论（不重走全部流程）
4. 辩论完成后，裁判更新方案修改日志
5. 如影响任务拆解，裁判更新 Step 6 任务清单
6. 裁决书新增"复庭记录"章节

```
## 复庭记录
| 复庭次数 | 日期 | 原因 | 影响 | 决议 |
|---------|------|------|------|------|
| 1 | [YYYY-MM-DD] | [...] | 修改方案/更新任务 | [决议] |
```

### 复庭命令

"复庭" | "复庭，原因是XX" | "针对XX重新辩论" | "方案已变更，重新评估可行性"
